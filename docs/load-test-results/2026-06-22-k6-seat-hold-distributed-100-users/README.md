# 분산 좌석 선점 100명 기준선 측정

## 조건

- 대상: 배포 EC2 `t4g.large`, HTTPS API Gateway 경유
- 시나리오: 연습 세션 생성 후 대기열 통과, 서로 다른 좌석 1개씩 선점
- 좌석 범위: 기본 연습 배치 첫 구역의 좌석 ID `1~2,500`
- 워밍업: 10명, 오픈 대기 5초, 1회 실행
- 본 측정: 100명, 오픈 대기 5초, 3회 실행
- 대기열 polling: 2초, heartbeat: 10초

모든 실행에서 100명 전원이 좌석 선점에 성공했고 HTTP 실패는 없었습니다.

## 내부 구간 p95 중앙값

| 구간 | 중앙값 (ms) | 해석 |
| --- | ---: | --- |
| entryToken 검증 | 21 | JWT 검증은 주요 병목이 아님 |
| 좌석 DB 검증 | 201 | DB 조회와 커넥션 획득 대기가 포함된 구간 |
| Redis Lua 선점 | 157 | 원자적 선점 처리 비용 |
| Ticket Service 선점 처리 전체 | 324 | 좌석 검증·Redis 선점·응답 모델 생성 포함 |
| API Gateway 선점 경로 | 850 | Gateway와 Ticket Service HTTP 경로 전체 비용 |

서로 다른 요청 표본의 p95이므로 각 내부 구간을 단순 합산해 전체 p95를 계산할 수는 없습니다.

## HikariCP 상태

| 지표 | 실행 1 | 실행 2 | 실행 3 | 해석 |
| --- | ---: | ---: | ---: | --- |
| active 최대 | 10 | 10 | 10 | 기본 풀 최대치를 모두 사용 |
| pending 최대 | 10 | 7 | 30 | DB 커넥션 획득 대기 발생 |

연습 모드 `validateLayoutSeats`는 선택 좌석을 조회한 뒤, 좌석마다 `existsByIdAndLayoutId`로 배치 소속을 다시 확인합니다. 현재 측정은 사용자당 1좌석이므로 좌석 검증마다 최소 2회의 DB 조회가 발생합니다. 다중 좌석 요청에서는 추가 확인 쿼리가 좌석 수만큼 늘어납니다.

따라서 다음 개선에서는 HikariCP 크기를 먼저 늘리기보다, 이미 조회한 좌석의 `section.layout` 정보로 소속을 검증해 추가 DB 조회를 제거하는 방안을 우선 검토합니다.
