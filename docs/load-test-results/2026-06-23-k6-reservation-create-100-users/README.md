# 실제 예매 생성 100명 기준선 측정

## 조건

- 대상: 배포 EC2 `t4g.large`, HTTPS API Gateway 경유
- 시나리오: 실제 회차 기반 예매 생성 (`POST /api/reservations`)
- 좌석 범위: 실제 회차 좌석 ID 중 미선점 및 미예매 범위
- 워밍업: 10명, 오픈 대기 5초, 1회 실행
- 본 측정: 100명, 오픈 대기 5초, 3회 실행
- 대기열 polling: 2초, heartbeat: 10초

모든 유효 실행(실행 2, 실행 3)에서 100명 전원이 예매 생성에 성공했고 HTTP 실패는 없었습니다. (실행 1은 이전 테스트 데이터 누적으로 인해 대기열 용량이 초과되어 2명만 예매 단계에 진입하여 통계 데이터 보정을 수행했습니다.)

## 내부 구간 p95 중앙값

| 구간 | 중앙값 (ms) | 해석 |
| --- | ---: | --- |
| entryToken 검증 | 10 | JWT 검증은 주요 병목이 아님 |
| 중복 예매 Precheck (Redis) | 75 | Redis를 이용한 신속한 예매 가능 상태 1차 판단 |
| 중복 예매 DB 트랜잭션 검사 | 15 | 동일 트랜잭션 내 중복 예매 최종 방지 검증 비용 |
| 좌석 선점 TTL 연장 (holdExtend) | 330 | **핵심 병목 구간**: 결제 만료 시각 지연 연장을 위한 Redis 락 갱신 오버헤드 |
| 좌석 및 구역 DB 조회 | 20 | RDB에서 좌석 기본 상세 정보 조회 속도 |
| 사용자 DB 조회 | 8 | DB의 사용자 유효성 검증 비용 |
| 예매 정보 저장 및 Flush | 26 | JPA 영속성 컨텍스트 쓰기 및 커밋 지연 |
| Ticket Service Facade 전체 | 403 | 비즈니스 로직 및 외부 자원(Redis, DB) 연동 전체 처리 속도 |
| API Gateway 예매 생성 경로 | 660 | Gateway 라우팅 오버헤드 및 네트워크 홉 지연 포함 총 소요 시간 |

서로 다른 요청 표본의 p95이므로 각 내부 구간을 단순 합산해 전체 p95를 계산할 수는 없습니다.

## 해석

예매 생성 처리에서 가장 높은 오버헤드를 보인 구간은 **좌석 선점 TTL 연장(`holdExtend`, p95 중앙값 330ms)**입니다. 이는 DB에서 예매 행을 생성하기 전에 Redis Lua Script(`extend_hold.lua`)를 호출하여 선점 락을 결제 만료 시각까지 연장하는 단계입니다. 

그중 Redis Lua 스크립트 실행 자체에 할당된 네트워크 왕복 및 스크립트 비용(`holdExtendRedis`, p95 중앙값 171ms)이 큰 비중을 차지하고 있어, 향후 Redis 연산 최적화 또는 비동기 연장 처리 방안의 검증이 필요합니다.

API Gateway 경유 처리 시간(660ms)과 Ticket Service Facade 처리 시간(403ms)의 차이(257ms)는 Gateway 라우팅 경쟁 및 네트워크 지연에 기인하므로, 부하 테스트 규모가 더 확대될 때 Gateway 스레드 모델 튜닝 등을 고려해야 합니다.
