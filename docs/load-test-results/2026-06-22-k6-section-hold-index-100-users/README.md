# 구역별 hold 인덱스 적용 후 사용자 100명 측정

## 목적

좌석 조회 시 2,500개 좌석 hold key를 Redis `MGET`하던 구조를 구역별 Sorted Set 인덱스 조회로 변경한 뒤,
Redis hold 조회 구간과 전체 예매 흐름의 응답 시간을 측정합니다.

## 환경과 절차

- 대상: 배포 EC2 `t4g.large`, API Gateway 경유 HTTPS 요청
- 시나리오: 연습 세션 생성부터 대기열, 좌석 조회·선점, 예매, 결제 완료까지
- 좌석 조회: 섹션당 2,500석, 전체 10,000석
- Prometheus: Spring 애플리케이션 scrape 주기 `1초`
- 워밍업: 사용자 100명, 티켓 오픈 5초 뒤 전체 예매 흐름 1회
- 안정화: 워밍업 완료 후 90초 대기, HikariCP pending `0` 확인
- 본 측정: 사용자 100명, 티켓 오픈 5초 뒤 전체 예매 흐름 3회
- Prometheus 수집 창: 티켓 오픈 뒤 30초

## hold 인덱스 조회 구간

| 지표 | 중앙값 | 이전 MGET 방식 중앙값 | 변화 |
| --- | ---: | ---: | ---: |
| hold 조회 전체 p95 | 175 ms | 356 ms | -181 ms (-50.8%) |
| Redis hold 조회 p95 | 175 ms | Redis `MGET` p95 289 ms | -114 ms (-39.4%) |
| hold 결과 매핑 p95 | 2.45 ms | 10 ms | -7.55 ms |
| Lettuce Lua 호출 평균 | 30 ms | Lettuce `MGET` 평균 65 ms | -35 ms |

구역별 인덱스 조회는 선점된 좌석 ID만 반환하므로, 2,500개 hold key 생성과 전체 `MGET`을 제거했습니다.
hold 조회 p95는 목표대로 감소했고, 결과 매핑 비용도 낮게 유지됐습니다.

이전 측정은 사용자 10명으로 워밍업했고, 이번 측정은 사용자 100명으로 워밍업했습니다.
따라서 전체 API 지표 개선 폭에는 hold 인덱스 변경뿐 아니라 워밍업 절차 개선 효과도 포함되어 있어,
전체 수치를 hold 인덱스만의 효과로 단정하지 않습니다.

## 해석

- 100명 워밍업 뒤 첫 실행이 유난히 느린 콜드 스타트 양상은 사라졌습니다.
- 다만 좌석 조회 p95는 `1.10초 → 1.23초 → 1.41초`로 실행 순서에 따라 증가했고, HikariCP pending 최대도 `36 → 66 → 68`로 증가했습니다.
  커넥션 대기가 순간적으로 `0`이 된 것만으로 전체 시스템이 동일한 상태로 돌아왔다고 볼 수는 없습니다.
- hold 인덱스는 Redis hold 조회 구간을 줄였고, 강화된 워밍업 조건에서 좌석 조회 p95도 `1.5초` 목표 안에 들어왔습니다.
- HikariCP active 최대는 세 실행 모두 `10`으로 풀 최대치를 사용했습니다. DB 사용 경로별 커넥션 점유 원인은 다음 개선에서 분리 계측이 필요합니다.
- 500명 이상 측정 전에는 HikariCP pending뿐 아니라 Kafka 소비 지연과 서비스 CPU·메모리까지 안정화 조건에 포함합니다.
