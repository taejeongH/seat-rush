# 응답 버퍼링 제거 후 좌석 조회 사용자 100명

## 목적

응답 버퍼링 제거가 좌석 조회 성능에 미치는 영향을 확인하기 위해 100명 예매 연습 흐름을 측정했습니다.
초기 JVM·커넥션·캐시 상태로 인한 왜곡을 줄이기 위해 실제 측정 전 동일 흐름을 한 번 실행했습니다.

## 환경과 절차

- 대상: 배포 EC2 `t4g.large`, API Gateway 경유 HTTPS 요청
- 시나리오: 연습 세션 생성부터 대기열, 좌석 조회·선점, 예매, 결제 완료까지
- 사전 부하: 실제 측정에서 제외한 100명 전체 흐름 1회
- 안정화: 사전 부하 완료 후 60초 대기
- 본 측정: 100명 동시 사용자, 3회 반복
- 좌석 조회 대상: 섹션당 2,500석, 전체 10,000석

| 실행 | Ticket Service 전체 (ms) | Repository (ms) | Redis hold 조회 (ms) | DTO 변환 (ms) | Gateway 전체 (ms) |
| --- | ---: | ---: | ---: | ---: | ---: |
| 1 | 1,683 | 445 | 463 | 4 | 2,320 |
| 2 | 1,636 | 262 | 324 | 3 | 1,648 |
| 3 | 2,127 | 332 | 353 | 4 | 2,144 |

## 해석

- Repository와 Redis hold 조회는 수백 ms 수준인데 Ticket Service 전체와 Gateway 전체 시간은 더 큽니다. 이후 단계에서 응답 직렬화, HTTP 전송, Gateway 구간을 추가 분석해야 합니다.
- 좌석 조회 결과 수가 많아 발생하는 DB 엔티티 로딩·영속성 컨텍스트 비용을 줄이기 위한 projection 개선은 현재 로컬 작업 브랜치에만 있고, 이 결과에는 아직 반영되지 않았습니다.
