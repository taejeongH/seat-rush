# 최적화 후 100명·500명·1,000명 예매 연습 단계별 측정 결과

## 목적

모든 개별 최적화 작업이 적용된 상태에서, 전체 예매 흐름(대기열 진입 ~ 결제 완료) 시나리오에 대해 100명, 500명, 1,000명 동시 사용자 부하를 주어 최적화 전후 성능을 종합적으로 평가하고 인프라/하드웨어 한계 지점을 분석합니다.

## 공통 조건

- 대상 환경: EC2 t4g.large, Docker Compose
- 부하 생성기: 로컬 PC의 k6
- 대상 주소: 배포 API Gateway
- 사용자 행동: 결제 성공 100%, 중도 이탈 0%
- 좌석 선택: 사용자당 1~4석, 선점 충돌 시 최대 5회 재시도
- 인증: 측정 시작 전 테스트 계정 토큰 갱신 후 동시 대기열 진입
- 실행 방식: 토큰 준비 완료 후 60초 대기, 같은 시각에 대기열 진입
- 워밍업: JVM JIT 컴파일 최적화 및 커넥션 풀 활성화를 위해 본 측정 전 50 VU 부하로 30~45초 동안 2회 사전 예열 단계를 수행하고, DB와 Redis 데이터를 초기화한 뒤 10초 대기 후 본 측정을 시작했습니다.

## 해석 및 한계 분석

### 1. JVM Warm-up (예열)을 통한 성능 정상화
- 6/23의 이전 측정에서 최적화 후 오히려 대기열 진입 및 예매 생성 레이턴시가 크게 상승했던 것은, 백엔드 서버가 새로 배포되거나 유휴 상태인 상황에서 JVM 클래스 로딩 및 JIT 컴파일이 진행되지 않았고, 커넥션 풀(HikariCP, Redis)이 예열되지 않은 Cold Start 상태로 본 테스트(100 VU)에 진입했기 때문이었습니다.
- 본 재측정에서는 본 부하를 주기 전에 50 VU 강도로 30초~45초 동안 실제 예매 흐름 시나리오를 2회 반복 실행하여 JVM 및 커넥션 풀을 충분히 예열하는 단계를 도입하였습니다. 그 결과, 모든 계측 데이터가 정상 범주로 수렴하였으며 최적화 이전 baseline 대비 성능 개선 효과가 명확하게 관측되었습니다.

### 2. 성능 최적화 효과 입증
- 개선된 코드가 제 기능을 발휘하면서 대량의 트래픽 상황일수록 최적화 효과가 더욱 두드러졌습니다.
- 500명 부하 기준, 좌석 조회는 2,822 ms에서 1,203 ms로 57.3% 단축되었고, 좌석 선점은 1,747 ms에서 910 ms로 47.9% 단축되었으며, 예매 생성은 2,543 ms에서 973 ms로 61.7% 단축되는 드라마틱한 성능 향상을 이뤄냈습니다.
- 1,000명 부하 기준에서도 좌석 조회는 3,324 ms에서 1,575 ms로 52.6% 단축되는 등 전 구간 레이턴시가 11.8% ~ 52.6%까지 고르게 향상되었습니다.

### 3. API Gateway의 자원 경합 및 1,000명 VUs 연결 한계
- 1,000명 부하 시 36건의 HTTP 실패(checks fail)가 관측되었습니다. (대기열 진입 실패 12건, queues/me 폴링 실패 22건, queues/enter 실패 2건)
- 이는 최적화 이전 baseline(44건) 대비 소폭 줄었으나 여전히 한계가 존재하는 상태입니다. 단일 인스턴스(t4g.large, 2 vCPU) 자원 내에서 Microservices와 모니터링 모듈이 동시에 구동되는 환경에서 Gateway(Nginx 및 Spring Cloud Gateway)가 스레드 경합으로 인해 일시적으로 연결을 거부하거나 타임아웃을 유발하는 구조적 임계점입니다. 향후 다중 인스턴스로의 분산 스케일아웃(Scale-out)이 추가적인 개선 방향이 될 수 있습니다.
