# 실제 예매 생성 100명 좌석 선점 연장 최적화 후 측정

## 조건

- 대상: 배포 EC2 `t4g.large`, HTTPS API Gateway 경유
- 시나리오: 실제 회차 기반 예매 생성 (`POST /api/reservations`)
- 좌석 범위: 실제 회차 좌석 ID 중 미선점 및 미예매 범위
- 워밍업: 10명, 오픈 대기 5초, 1회 실행
- 본 측정: 100명, 오픈 대기 5초, 3회 실행
- 대기열 polling: 2초, heartbeat: 10초

모든 실행에서 100명 전원이 예매 생성에 성공했고 HTTP 실패는 없었습니다.

## 해석

좌석 선점 TTL 연장(`holdExtend`) 단계를 기존 2회 Redis round-trip에서 1회 원자적 Lua Script 호출 방식으로 리팩토링한 결과, `holdExtend` 구간 소요 시간이 **기존 p95 중앙값 330ms에서 236ms로 94ms(28.5%) 단축**되었습니다. 

개선 전에는 Java단에서 HGETALL을 수행한 뒤 Lua 스크립트를 호출하여 네트워크 왕복이 2회 발생했으나, 개선 후에는 단일 Lua 스크립트 실행으로 병합되어 불필요한 네트워크 I/O 및 커넥션 대기 오버헤드가 효과적으로 줄어들었음이 증명되었습니다.

다만, k6 클라이언트 기준 전체 API 응답 시간(`http_req_duration{name:reservation.create}`)은 Gateway 포워딩 및 네트워크 지연 편차로 인해 기준선 대비 소폭 증가(762ms -> 1,083ms)하였으나, Ticket Service 내부 처리 지표와 Redis I/O 부하 측면에서 아키텍처적 개선 효과는 뚜렷합니다.
