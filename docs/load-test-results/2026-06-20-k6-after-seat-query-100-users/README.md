# 좌석 조회 개선 후 k6 재측정 결과: 100명

## 실행 조건

| 항목 | 값 |
| --- | --- |
| 대상 환경 | EC2 `t4g.large`, Docker Compose |
| 부하 생성기 | 로컬 PC의 k6 |
| 대상 주소 | 배포 API Gateway |
| 좌석 배치 | 연습 배치 ID `1`, 전체 10,000석, 구역당 2,500석 |
| 사용자 행동 | 결제 성공 100%, 중도 이탈 0% |
| 좌석 선택 | 사용자당 1~4석, 선점 충돌 최대 5회 재시도 |
| 워밍업 | 각 본 측정 전 10명 전체 예매 흐름 실행 후 30초 대기 |
| 계정 토큰 준비 | 본 측정 전 100개 계정을 동시 로그인해 access token 준비 |
| 반복 횟수 | 100명 본 측정 3회 |

## 개선 전 비교

| 항목 | 개선 전 | 개선 후 중앙값 | 변화 |
| --- | ---: | ---: | ---: |
| 좌석 조회 p95 | 2,180 ms | 3,237 ms | +1,057 ms (+48.5%) |

개선 전 결과는 1회 측정값이고, 개선 후 결과는 3회 중앙값입니다. 따라서 이 수치만으로 조회
쿼리 변경이 성능을 악화시켰다고 단정하지 않습니다. 다만 현재 조건에서 외부 좌석 조회 지연이
여전히 크므로, 500명 이상 측정 전에 응답 직렬화와 Gateway 구간을 추가로 분석합니다.

## 좌석 조회 구간별 p95

새로 추가한 `seat.query.*` 메트릭을 실행 시간 창별로 조회했습니다.

| 구간 | 실행 1 | 실행 2 | 실행 3 | 중앙값 |
| --- | ---: | ---: | ---: | ---: |
| `seat.query` | 626 ms | 1,138 ms | 838 ms | 838 ms |
| `seat.query.static` | 409 ms | 683 ms | 645 ms | 645 ms |
| `seat.query.hold.read` | 414 ms | 445 ms | 441 ms | 441 ms |
| `seat.query.mapping` | 10 ms | 5 ms | 7 ms | 7 ms |

중앙값 기준 Ticket Service의 `seat.query` p95는 약 838ms였지만 외부 k6 좌석 조회 p95는
3.24초였습니다. DB·Redis·DTO 변환 외에 HTTP 응답 직렬화, Gateway 대기, 응답 전송 구간의
비중을 계속 확인해야 합니다.

## 결론

- 100명 모두 예매를 완료했고 HTTP 요청 실패는 없었습니다.
- 좌석 조회의 DB·Redis·DTO 변환 중 DTO 변환은 병목이 아니었습니다.
- 500명·1,000명 확대 측정은 보류합니다. 먼저 큰 좌석 응답의 직렬화·전송 비용과 Gateway 경로를
  계측한 뒤 동일 100명 조건으로 다시 확인합니다.
