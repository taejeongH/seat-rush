# 실제 예매 생성 100명 중복 예매 검증 제거 최적화 후 측정

## 조건

- 대상: 배포 EC2 `t4g.large`, HTTPS API Gateway 경유
- 시나리오: 실제 회차 기반 예매 생성 (`POST /api/reservations`)
- 좌석 범위: 실제 회차 좌석 ID 중 미선점 및 미예매 범위
- 워밍업: 10명, 오픈 대기 5초, 1회 실행
- 본 측정: 100명, 오픈 대기 5초, 3회 실행
- 대기열 polling: 2초, heartbeat: 10초

모든 실행에서 100명 전원이 예매 생성에 성공했고 HTTP 실패는 없었습니다.

| 구간 | 중앙값 (ms) | 해석 |

| 사용자 DB 조회 | 34 | DB의 사용자 유효성 검증 비용 |

## 해석

예매 생성 API 호출 과정에서 비즈니스 로직 및 트랜잭션 내부에서 매번 수행되던 중복 예매 검증용 DB SELECT 조회(existsByHoldId)를 완전히 걷어내고, MySQL의 unique 인덱스 제약 조건을 기반으로 예외 처리를 유도하는 Optimistic Insert 방식으로 구조를 단순화했습니다.

그 결과 Ticket Service Facade의 p95 소요 시간이 기존 522ms에서 418ms로 104ms(19.9%) 단축되었습니다. DB SELECT 쿼리가 트랜잭션 안팎에서 총 2회 줄어듦에 따라 HikariCP의 커넥션 점유 시간이 효과적으로 단축되었으며, 이에 따라 pending 커넥션 최대값도 기존 30개에서 17개 수준으로 대폭 완화되었습니다.

Gateway 및 네트워크 변수에 따른 편차를 제외하고, Ticket Service 내부 처리량 기준 성능 병목이 획기적으로 개선되었음이 입증되었습니다.
