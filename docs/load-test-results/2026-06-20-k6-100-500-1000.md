# k6 예매 연습 단계별 측정 결과

## 목적

로그인과 계정 준비가 끝난 사용자가 같은 시각에 대기열에 진입하는 티켓 오픈 상황을 기준으로,
100명부터 1,000명까지 전체 예매 흐름의 응답 시간과 오류율을 확인합니다.

이 문서는 **좌석 조회 개선 전 단계별 기준선**입니다. 이후 개선 결과는 별도 문서에서 동일 조건으로 비교합니다.

## 공통 조건

| 항목 | 값 |
| --- | --- |
| 대상 환경 | EC2 `t4g.large`, Docker Compose |
| 부하 생성기 | 로컬 PC의 k6 |
| 대상 주소 | 배포 API Gateway |
| 사용자 행동 | 결제 성공 100%, 중도 이탈 0% |
| 좌석 선택 | 사용자당 1~4석, 선점 충돌 시 최대 5회 재시도 |
| 인증 | 측정 시작 전 테스트 계정 토큰 20개씩 갱신 |
| 실행 방식 | 토큰 준비 완료 후 60초 대기, 같은 시각에 대기열 진입 |
| 워밍업 | 각 본 측정 전 10명 5초 카운트다운 실행 후 15초 대기 |

좌석 조회는 연습 배치 ID `1`의 전체 10,000석을 사용했습니다. 4개 구역은 각각 2,500석이며,
각 사용자는 선택한 구역의 2,500석 전체를 조회합니다. Ticket Service는 좌석 정보를 DB에서 조회한 뒤,
동일한 좌석 ID 목록을 기준으로 Seat Redis의 hold 상태를 조회해 응답을 조합합니다.

## 전체 결과

| 동시 사용자 | 예매 확정 | HTTP 실패율 | 전체 HTTP p90 | 전체 HTTP p95 | 전체 HTTP p99 | 좌석 충돌 재시도 | 판정 |
| ---: | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| 100 | 100 / 100 | 0.000% | 1,224 ms | 1,527 ms | 2,108 ms | 2 | 정상 완료, 응답 시간 개선 필요 |
| 500 | 500 / 500 | 0.000% | 1,204 ms | 1,486 ms | 2,435 ms | 4 | 정상 완료, 응답 시간 개선 필요 |
| 1,000 | 956 / 1,000 | 0.157% | 2,281 ms | 2,670 ms | 3,681 ms | 17 | `queues/me` 연결 종료 오류 발생 |

1,000명에서는 일부 `queues/me` 요청이 `EOF` 또는 연결 종료로 실패했습니다. 44명은 예매 확정까지 도달하지 못했고,
단일 EC2 구성의 첫 한계 지점으로 기록합니다.

## 해석과 후속 측정

- 500명까지는 전체 예매가 완료됐지만, 입장 토큰 발급·좌석 조회·예매 생성 p95가 목표를 넘었습니다.
- 1,000명에서는 API 지연뿐 아니라 Gateway 또는 하위 서비스 연결 종료가 발생했습니다.
- 좌석 조회는 단순 DB SELECT 시간만 의미하지 않습니다. 2,500개 좌석 DB 조회, Redis hold 상태 조회, DTO 조합, JSON 직렬화와 Gateway 전송 시간을 포함합니다.
- 응답 버퍼링 제거 후 좌석 조회 재측정은 [응답 버퍼링 제거 후 좌석 조회 사용자 100명](./2026-06-21-k6-response-buffer-removed-100-users/README.md)에서 관리합니다.
- projection 적용 후에는 이 기준선과 별도 재측정 결과를 비교합니다.
