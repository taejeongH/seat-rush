# 연습 좌석 검증 조회 제거 후 분산 좌석 선점 100명 재측정

## 조건

- 대상: 배포 EC2 `t4g.large`, HTTPS API Gateway 경유
- 시나리오: 연습 세션 생성 후 대기열 통과, 서로 다른 좌석 1개씩 선점
- 좌석 범위: 기본 연습 배치 첫 구역의 좌석 ID `1~2,500`
- 워밍업: 10명, 오픈 대기 5초, 1회 실행
- 본 측정: 100명, 오픈 대기 5초, 3회 실행
- 변경: 연습 좌석 검증 시 좌석별 `existsByIdAndLayoutId` 조회 제거

모든 실행에서 100명 전원이 좌석 선점에 성공했고 HTTP 실패는 없었습니다.

## 해석

연습 좌석 검증에서 이미 조회한 `section.layout` 정보로 배치 소속을 비교하도록 변경하면서, 좌석별 `existsByIdAndLayoutId` DB 조회를 제거했습니다.

좌석 DB 검증 p95와 Ticket Service 선점 처리 p95가 함께 감소했고, HikariCP 커넥션 대기도 거의 사라졌습니다. Redis Lua 선점 시간도 감소했지만, Redis 구현을 변경하지 않았으므로 DB 대기 감소에 따른 간접 효과로 해석합니다.

실행 2는 다른 실행보다 높은 p95를 보였지만, 3회 중앙값 기준으로 모든 핵심 지표가 개선됐습니다.
