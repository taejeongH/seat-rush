# 최적화 후 100명 ~ 3,000명 예매 연습 단계별 측정 결과

## 목적

애플리케이션 및 인프라 전반의 모든 기능 최적화(JPA Projection, Redis 인덱싱, 대기열 만료 로직 개선 등)와 OS 커널 및 리슨 소켓 네트워크 백로그 튜닝 적용 후의 종합 성능을 검증합니다. 최초 개선 전(6/20)과 모든 최적화가 적용된 최종 개선 후(6/24)의 성능을 동시 사용자 규모별로 비교 분석합니다.

## 공통 조건

- 대상 환경: EC2 t4g.large, Docker Compose
- 부하 생성기: 로컬 PC의 k6
- 대상 주소: 배포 API Gateway
- 사용자 행동: 결제 성공 100%, 중도 이탈 0%
- 좌석 선택: 사용자당 1~4석, 선점 충돌 시 최대 5회 재시도
- 인증: 측정 시작 전 테스트 계정 토큰 갱신 후 동시 대기열 진입
- 실행 방식: 토큰 준비 완료 후 60초 대기, 같은 시각에 대기열 진입
- 워밍업: 본 측정 전 50명(가상 사용자) 부하로 예열 단계를 수행하고, DB와 Redis 데이터를 초기화한 뒤 본 측정을 시작했습니다.

## 결과 해석 및 임계 분석

### 1. 백로그 설정 조정에 의한 수용력 증대 효과
- 1,000명 상황에서 빈발하던 Nginx 및 Gateway의 무작위 커넥션 단절(EOF/RST)을 차단하여 0.00% 오류율로 100% 예약 처리를 완료했습니다.
- 백로그가 늘어남에 따라 순간 드랍되던 커넥션들이 큐에서 지연을 감내하며 순차적으로 성공(예매 확정) 단계로 진입하여, 3,000명 환경에서의 오류 건수 또한 대폭 완화되었습니다.

### 2. 단일 인스턴스 환경의 컴퓨팅 포화
- 3,000명 부하에서 나타난 7.74%의 실패는 Nginx와 애플리케이션 백로그 큐를 최대로 늘렸음에도 불구하고, 단일 가상 머신(t4g.large)의 2 vCPU 하드웨어가 3,000개 동시 요청의 스레드 경합(Load Average 3.0 초과)을 물리적으로 처리해내지 못해 발생한 소켓 처리 시간 초과 드랍입니다.
- 단일 서버의 인프라적 성능 튜닝은 임계 도달하였으므로, 향후 성능을 더 높이기 위해서는 API Gateway 및 Core 서비스를 다중 인프라로 분리하는 스케일아웃(Scale-out)을 검토해야 합니다.
